# Provenance Guard — Planning

An attribution-analysis backend that classifies whether submitted text was written by a
human or generated by AI, scores its confidence honestly, surfaces a plain-language
transparency label, and lets creators appeal a classification they believe is wrong.

The guiding principle for every design decision below: **on a writing platform, wrongly
labeling a human's work as AI-generated is worse than missing AI.** A false accusation
damages a real creator's reputation and trust; a missed detection is a softer failure.
That asymmetry shows up in our thresholds, our label wording, and our prominent appeal path.

---

## Architecture

Two flows: a **submission flow** (text in, classification + label out) and an **appeal
flow** (creator contests a result). Every decision and every appeal is written to an
append-only audit log.

```
SUBMISSION FLOW
===============

  POST /submit
  { text, creator_id }
        |
        |  raw text
        v
  +------------------------+          +------------------------------+
  | Signal 1: LLM (Groq)   |          | Signal 2: Stylometrics       |
  | llama-3.3-70b-versatile |          | pure Python heuristics       |
  | semantic / holistic     |          | structural / statistical     |
  +------------------------+          +------------------------------+
        |                                    |
        | llm_score in [0,1]                 | stylo_score in [0,1]
        | (= P(AI))                          | (= P(AI))
        v                                    v
  +--------------------------------------------------------------+
  | Confidence scoring                                           |
  | confidence = 0.6 * llm_score + 0.4 * stylo_score            |
  +--------------------------------------------------------------+
        |
        | confidence in [0,1]
        v
  +--------------------------------------------------------------+
  | Label mapping (threshold bands)                             |
  |   < 0.40            -> likely_human                         |
  |   0.40 .. 0.65      -> uncertain                            |
  |   > 0.65            -> likely_ai                            |
  +--------------------------------------------------------------+
        |
        | attribution + label text
        v
  +--------------------------------------------------------------+
  | Audit log (append)  content_id, creator_id, timestamp,      |
  | attribution, confidence, llm_score, stylo_score,            |
  | status="classified"                                         |
  +--------------------------------------------------------------+
        |
        | content_id, attribution, confidence, label
        v
  RESPONSE { content_id, attribution, confidence, label }


APPEAL FLOW
===========

  POST /appeal
  { content_id, creator_reasoning }
        |
        |  content_id (locate original decision)
        v
  +--------------------------------------------------------------+
  | Look up original classification in `content` table          |
  +--------------------------------------------------------------+
        |
        |  set status
        v
  +--------------------------------------------------------------+
  | Update content.status -> "under_review"  (in place)         |
  +--------------------------------------------------------------+
        |
        |  append (never overwrite the original decision)
        v
  +--------------------------------------------------------------+
  | Audit log (append) appeal event: content_id, timestamp,     |
  | creator_reasoning, references original attribution,         |
  | status="under_review"                                       |
  +--------------------------------------------------------------+
        |
        v
  RESPONSE { content_id, status: "under_review", message }
```

**Narrative.** A submission's raw text is scored independently by two distinct signals —
a semantic LLM classifier and a set of structural stylometric heuristics — whose outputs
are combined into a single calibrated confidence score. That score is mapped to one of
three transparency labels and written, with both individual signal scores, to an
append-only audit log before the response is returned. An appeal locates the original
decision by `content_id`, flips that content's status to "under review," and appends a new
audit event recording the creator's reasoning alongside (never replacing) the original
classification. No automated re-classification occurs.

## Detection signals

We use two **distinct** signals: one measures *meaning*, the other measures *structure*.
They fail on different inputs, which is what makes the pair more informative than either
alone. Two LLM prompts would both be semantic and would fail together — not distinct.

### Signal 1 — LLM classification (Groq, `llama-3.3-70b-versatile`)

- **Measures:** whether the text *reads* as human or AI-generated, judged holistically —
  semantic coherence, register, the tell-tale evenness and "hedged completeness" of AI prose.
- **Output:** a single float `llm_score` in `[0,1]` = the model's estimated probability the
  text is AI-generated. We prompt for strict JSON (`{"ai_probability": 0.0-1.0, "reasoning": "..."}`)
  and parse it.
- **Why it differs between human and AI:** current LLMs produce fluent, evenly-developed,
  low-risk prose; humans are messier, more idiosyncratic, more uneven.
- **Blind spot:** LLM-as-detector is unreliable and **biased toward false positives on
  formal, polished, or non-native-English human writing**, and it is non-deterministic.
  This is the single biggest risk to a real creator, and it drives our whole asymmetry design.

### Signal 2 — Stylometric heuristics (pure Python)

Three measurable properties, combined into one `stylo_score` in `[0,1]`:

- **Sentence-length variance ("burstiness"):** humans vary sentence length a lot; AI text is
  more uniform. Low variance -> more AI-like.
- **Type-token ratio (lexical diversity):** vocabulary richness over a fixed window (TTR is
  length-sensitive, so we cap the window to keep it comparable across inputs).
- **Punctuation variety/density:** humans use a wider, less regular mix of punctuation.

Each metric is normalized to a 0-1 "AI-likeness" contribution and averaged into `stylo_score`.

- **Why it differs between human and AI:** these are statistical fingerprints of *how* text is
  assembled, independent of what it means.
- **Blind spot:** **structurally uniform human writing scores as AI** — poetry with heavy
  repetition, formal/academic prose, and very short texts (too few sentences to measure
  variance reliably). See Edge Cases.

### Combining the signals

`confidence = 0.6 * llm_score + 0.4 * stylo_score`

The LLM is the stronger holistic signal, so it carries more weight; stylometrics acts as an
independent structural cross-check that can pull an over-confident LLM back toward the middle.

## Uncertainty representation

**What the score means:** `confidence` is `P(AI-generated)` on a single axis. `0` means we are
confident the text is human; `1` means we are confident it is AI; **`0.5` means maximally
uncertain — the system is telling the user it genuinely cannot decide.** "Confidence" in the
everyday sense (how *sure* we are) is the distance from 0.5, not the raw number. This is why
`0.51` and `0.95` must produce different labels: `0.51` is barely-off-the-fence (uncertain),
`0.95` is a strong AI signal.

**Mapping raw outputs to a calibrated score:** both signals already emit `[0,1]` = P(AI), so
the weighted average is directly interpretable; no separate calibration curve is needed for
the MVP. We validate calibration empirically (below) rather than assuming it.

**Thresholds (asymmetric on purpose):**

| Combined confidence | Attribution     | Rationale |
|---------------------|-----------------|-----------|
| `< 0.40`            | `likely_human`  | Clear the creator readily. |
| `0.40 – 0.65`       | `uncertain`     | Default band when signals are weak or disagree. |
| `> 0.65`            | `likely_ai`     | Require *stronger* evidence before accusing. |

The AI cutoff (0.65) sits **0.15 above** the neutral midpoint (0.5); the human cutoff (0.40)
sits only **0.10 below** it. We deliberately demand more evidence to call something AI than to
clear it, so a borderline case defaults to the less-harmful outcome — the false-positive
asymmetry, made concrete.

**How we test that scores are meaningful:** run four deliberately chosen inputs — clearly AI,
clearly human, formal-human (borderline), lightly-edited-AI (borderline) — log both individual
signal scores and the combined score, and confirm (a) clearly-AI scores well above clearly-human,
and (b) the two borderline cases land in the uncertain band rather than flipping to an extreme.
Two of these, with real numbers, go in the README.

## Transparency label design

Plain language, meaningful to a non-technical reader, and — for any AI result — hedged and
paired with the appeal path. Exact text of all three variants:

**`likely_human` (confidence `< 0.40`)**
> ✓ **Likely human-written.** Our analysis found writing patterns consistent with human
> authorship, and no strong signals of AI generation. This is an automated estimate, not a
> guarantee.

**`uncertain` (confidence `0.40 – 0.65`)**
> ● **Origin uncertain.** Our analysis was inconclusive — this content shows a mix of signals,
> and we can't determine with confidence whether it was written by a person or generated by AI.
> Please treat this result as informational only.

**`likely_ai` (confidence `> 0.65`)**
> ⚠ **Signals consistent with AI generation.** Our analysis found patterns often associated with
> AI-generated text. This is not a definitive judgment, and automated detection can be wrong —
> especially for formal or non-native-English writing. If you wrote this yourself, you can appeal.

The AI variant never says "this was made by AI"; it says "signals consistent with," names its own
known weakness, and offers the appeal — the asymmetry expressed in UX.

## Appeals workflow

- **Who can appeal:** the creator of a submission, identified by the `content_id` returned at
  submission time.
- **What they provide:** `content_id` and `creator_reasoning` (free text explaining why they
  believe the classification is wrong).
- **What the system does on receipt:**
  1. Locate the original decision by `content_id`.
  2. Update that content's status **in place** to `under_review`.
  3. **Append** a new audit-log event recording the appeal — timestamp, `creator_reasoning`, and a
     reference to the original attribution/confidence. The original classification entry is never
     modified or deleted.
  4. Return a confirmation that the appeal was received.
- **No automated re-classification.** A human reviewer would later open the appeal queue and see:
  the original text, the original attribution + confidence + both signal scores, and the creator's
  reasoning, side by side.

The append-only property matters: the audit log is the evidentiary record. Overwriting the
original decision would erase the very thing the creator is contesting.

## Anticipated edge cases

1. **Repetitive/simple poetry or lyrics.** Heavy repetition and small vocabulary drive
   sentence-length variance and type-token ratio down, so the *stylometric* signal scores it
   AI-like — a false positive against a human creative work. Mitigation: the LLM signal often
   recognizes it as creative human writing, and when the two signals disagree the combined score
   stays in the uncertain band rather than flipping to `likely_ai`.
2. **Formal, academic, or non-native-English human writing** (e.g. a paragraph on monetary
   policy). Uniform, complex, formal prose reads "AI-ish" to *both* signals — this is exactly the
   demographic most harmed by a false positive. Mitigation: the asymmetric `> 0.65` AI threshold,
   the hedged AI label, and the prominent appeal path.
3. **Very short submissions** (fewer than ~3 sentences). Too little text to compute stylometric
   variance meaningfully, so `stylo_score` is unreliable; the combined result should be treated as
   low-confidence and will typically land in the uncertain band.

**M3 — submission endpoint + first signal**

  (Groq LLM) function returning `{"ai_probability", "reasoning"}`.

  wiring it into the endpoint; confirm `/submit` returns `content_id`, `attribution`, `confidence`,
  `label` and writes a structured audit entry.

**M4 — second signal + confidence scoring**
- *Provide:* the full Detection Signals section + Uncertainty Representation (thresholds) + diagram.
- *Ask for:* the Signal 2 (stylometrics) function and the confidence-scoring/label-mapping logic.
- *Verify:* run the four calibration inputs; confirm the combined score matches the specified
  thresholds (AI tools sometimes drift from the stated ranges — check, don't trust); confirm
  clearly-AI and clearly-human land far apart and borderlines land in the middle. Log both
  individual scores plus the combined score.

**M5 — production layer**
- *Provide:* Transparency Label variants + Appeals Workflow + diagram.
- *Ask for:* the label-generation function mapping confidence -> exact label text, the
  `POST /appeal` endpoint, `Flask-Limiter` config, and the `GET /log` endpoint.
- *Verify:* all three label variants are reachable via inputs at different confidence levels; an
  appeal flips status to `under_review` and appears in `GET /log`; the rate limiter returns `429`
  after the per-minute limit.

## Rate limiting

`10 per minute; 100 per day`, keyed by client IP.

- *10/minute* accommodates a real writer publishing a batch (e.g. a poetry collection) in one
  sitting, while stopping rapid-fire flooding.
- *100/day* caps sustained scripted abuse — a bot probing the classifier would exceed this quickly.
- *Keyed by IP* because there is no auth in this prototype. **Limitation:** IP-keying is crude
  (shared IPs, VPNs, NAT); in production we would key by authenticated `creator_id`.
